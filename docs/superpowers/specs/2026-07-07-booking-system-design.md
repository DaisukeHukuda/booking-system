# 自社専用予約管理システム 設計書

作成日: 2026-07-07
ステータス: 承認済み（ブレインストーミング完了）

## 1. 概要

アソビュー社の予約管理システム「ウラカタ予約」（https://biz.asoview.com/urakata-yoyaku）をモデルにした、自社専用の予約管理システムを開発する。中核はカレンダー型のWeb予約台帳・コース連動（在庫一括制御）・代理店連携の3つ。

### スコープ

- **対象ユーザー**: 自社スタッフ（管理画面）と外部代理店（専用リンク）。一般客向け公開予約ページは対象外
- **決済**: システム内オンライン決済はスコープ外。支払方法・支払状況の記録のみ行い、将来のStripe組み込みに耐えるデータ設計にする
- **予算**: インフラは無料枠のみ（独自ドメイン代を除く）

## 2. 技術スタック

| 項目 | 選定 | 理由 |
|---|---|---|
| 実行環境 | Cloudflare Workers（無料枠） | 商用利用可・スリープなし・10万リクエスト/日 |
| フレームワーク | Hono（TypeScript） | Workers最適化、JSXによるSSRで構成が単純 |
| DB | Cloudflare D1（SQLite） | 無料5GB。マイグレーションは wrangler d1 migrations |
| 画面 | Hono JSX による SSR ＋ 最小限の素のJS | SPA不採用。ビルド・テスト・デプロイを単純に保つ |
| メール | Resend（無料 100通/日） | 予約・キャンセル通知 |
| テスト | vitest + @cloudflare/vitest-pool-workers | ローカルでD1込みの実環境同等テスト |

- タイムゾーンは Asia/Tokyo 固定。日付は `YYYY-MM-DD` 文字列で保存し、UTC変換バグを構造的に排除する
- 管理者認証: 管理者パスワード1つ＋署名付きセッションCookie
- 代理店認証: `/a/{32文字ランダムトークン}` の専用URL。ログイン不要。トークンは再発行（＝旧リンク無効化）可能

## 3. データモデル

| テーブル | 主な列 | 役割 |
|---|---|---|
| `plans` | id, name, description, price, active, sort_order | プラン |
| `resources` | id, name, active | 共有リソース（インストラクター等） |
| `plan_resources` | plan_id, resource_id | プランが消費するリソース（多対多） |
| `slot_types` | id, name, start_time, sort_order | 時間帯マスタ（例: 午前便 9:00 / 午後便 13:00） |
| `plan_slots` | plan_id, slot_type_id, capacity, active | プラン×時間帯の催行設定と定員 |
| `slot_closures` | date, slot_type_id, plan_id(null=全プラン), reason | 手動クローズ（休業日等） |
| `bookings` | id, plan_id, date, slot_type_id, agency_id(null=自社), status, customer_name, customer_phone, party_size, total_amount, payment_method, payment_status, notes, created_by, created_at, cancelled_at | 予約本体 |
| `agencies` | id, name, token(unique), email, active, notes | 代理店 |
| `email_log` | id, booking_id, to, type, status, error, created_at | 通知メールの送信記録 |

- `bookings.status` は confirmed / cancelled（キャンセルは論理削除で記録が残る）
- `payment_method` は onsite_cash / onsite_card / invoice / stripe を想定した文字列列。`payment_status` は unpaid / paid。Stripe導入時にスキーマ変更が不要な形にしておく

## 4. コア在庫ロジック（コース連動）

「自動クローズ/オープン」を状態として保存せず、空き状況を予約データからの**導出値**として毎回計算する。

> プランPの日付D・時間帯Sが予約可能 ⇔
> ① Pが時間帯Sに催行設定されている（plan_slots.active）
> ② (D, S, P) が手動クローズされていない（全プラン指定のクローズも含む）
> ③ Pの (D, S) における confirmed 予約の人数合計 ＋ 新規人数 ≦ 定員
> ④ Pが必要とする全リソースについて、同じ (D, S) でそのリソースを必要とする**他プラン**の confirmed 予約が存在しない（同一プランの予約は定員まで相乗り可）

- クローズ/オープンの更新処理そのものが存在しないため、状態の不整合が原理的に起きない。キャンセルすれば次の計算で自動的にオープンに見える
- **ダブルブッキング対策**: 予約登録は、上記の空き条件をWHERE句に含めた `INSERT ... SELECT ... WHERE <空き条件>` を1文で実行し、`meta.changes` で成否を判定する。D1のバッチ実行はアトミックであるため、同時アクセスでも定員超過・リソース競合は発生しない
- 競合で登録できなかった場合は「この枠は直前に埋まりました」という明確なエラーを利用者に返す
- **予約の変更**（日付・時間帯・人数の変更）は「新枠での空き検証付き登録＋旧予約のキャンセル」を1バッチでアトミックに実行する。新枠が取れない場合は元の予約を維持したままエラーを返す
- 管理画面からの登録も代理店と同じ空き条件に従う（空き条件を無視した強制登録機能は設けない。例外運用は手動クローズの解除やマスタ変更で行う）

## 5. 画面構成

### 管理画面（`/admin`、パスワードログイン）

- **予約台帳カレンダー（ホーム）**: 月表示。各日に時間帯ごとの状態を色分け表示 — 空きあり（残席数）/ 満席 / 連動クローズ（原因プラン表示付き）/ 手動クローズ。日クリックで当日の予約一覧＋新規予約フォーム
- **予約の登録・変更・キャンセル**: 電話予約等もここから手入力（在庫連動は同様に働く）
- **マスタ管理**: プラン / リソース / 時間帯 / プラン別定員 / 手動クローズ日
- **代理店管理**: 登録、専用リンクの発行・再発行
- **CSVエクスポート**: 期間指定で予約データを出力。既存の月次売上集計フロー（reservations.csv）で扱える列構成にする

### 代理店ページ（`/a/{token}`、ログイン不要）

- **空きカレンダー**: 向こう60日のプラン別空き状況（残席数付き）。連動クローズも自動反映
- **予約フォーム**: プラン・日付・時間帯・人数・顧客名・電話番号・備考
- **自店予約一覧**: 自店の予約の確認とキャンセルのみ（他店・自社の予約は不可視）。キャンセル期限は当面設けない（開催日当日までキャンセル可。将来必要になれば設定項目として追加）

## 6. 通知とエラー処理

- 予約成立・キャンセル時に自社メールへ通知。代理店にメールアドレスが登録されていれば控えも送信
- 予約確定とメール送信は分離し、送信失敗で予約が失敗しない。結果は `email_log` に記録し、失敗は管理画面に警告表示

## 8. テストと開発体制

- **TDD**: 在庫ロジックは「連動クローズ」「キャンセルによる自動オープン」「同時予約の競合」「定員境界」「手動クローズとの組合せ」を網羅するテストを先に書く
- **体制**: メインセッション（Fable 5）は設計・レビュー・監査および在庫コアロジックの実装を担当。画面・CRUD等の定型実装はSonnetサブエージェント、軽微な修正はHaikuに委譲する
- **Git**: ローカルで管理を開始し、GitHubリポジトリ提供後にpushする
