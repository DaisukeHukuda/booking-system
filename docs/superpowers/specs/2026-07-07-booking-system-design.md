# 自社専用予約管理システム 設計書

作成日: 2026-07-07
改訂: 2026-07-07 v2 — 実際のウラカタ運用スクリーンショットに基づくモデル拡張（時間重なり連動・日別定員・リクエスト予約・料金内訳）

## 1. 概要

アソビュー社の予約管理システム「ウラカタ予約」（https://biz.asoview.com/urakata-yoyaku）をモデルにした、自社専用の予約管理システム。中核はカレンダー型のWeb予約台帳・コース連動（在庫一括制御）・代理店連携の3つ。

### スコープ

- **対象ユーザー**: 自社スタッフ（管理画面）と外部代理店（専用リンク）。一般客向け公開予約ページは対象外
- **決済**: システム内オンライン決済はスコープ外。支払方法・支払状況の記録のみ行い、将来のStripe組み込みに耐えるデータ設計にする
- **予算**: インフラは無料枠のみ（独自ドメイン代を除く）

### v2で追加した要件（実運用スクショより）

1. **時間の重なり連動**: プランごとに複数の開始時刻（例: 10:00/10:30/11:00）があり、所要時間が重なる予約同士はリソースを取り合う
2. **日別定員**: 同じプラン・時刻でも日によって定員が変わる（例: 平日7名/週末20名）
3. **リクエスト予約**: 代理店からの予約を「承認待ち」で受け、自社が承認/否認するフロー（代理店ごとに即確定/リクエストを設定）
4. **料金内訳**: 大人/小人の人数・単価を記録

## 2. 技術スタック

| 項目 | 選定 | 理由 |
|---|---|---|
| 実行環境 | Cloudflare Workers（無料枠） | 商用利用可・スリープなし・10万リクエスト/日 |
| フレームワーク | Hono（TypeScript） | Workers最適化、JSXによるSSRで構成が単純 |
| DB | Cloudflare D1（SQLite） | 無料5GB。マイグレーションは wrangler d1 migrations |
| 画面 | Hono JSX による SSR ＋ 最小限の素のJS | SPA不採用。ビルド・テスト・デプロイを単純に保つ |
| メール | Resend（無料 100通/日） | 予約・キャンセル・リクエスト通知 |
| テスト | vitest + @cloudflare/vitest-pool-workers | ローカルでD1込みの実環境同等テスト |

- タイムゾーンは Asia/Tokyo 固定。日付は `YYYY-MM-DD` 文字列で保存し、UTC変換バグを構造的に排除する
- 管理者認証: 管理者パスワード1つ＋署名付きセッションCookie
- 代理店認証: `/a/{32文字ランダムトークン}` の専用URL。ログイン不要。トークンは再発行（＝旧リンク無効化）可能

## 3. データモデル（v2）

| テーブル | 主な列 | 役割 |
|---|---|---|
| `plans` | id, name, description, **price_adult, price_child, duration_min**, active, sort_order | プラン。所要時間（分）は重なり判定に使用 |
| `resources` | id, name, active | 共有リソース（インストラクター等） |
| `plan_resources` | plan_id, resource_id | プランが消費するリソース（多対多） |
| `slot_types` | id, name, start_time, sort_order | **開始時刻マスタ**（例: 09:00 / 10:30 / 13:00。全プラン共通のリストから各プランが使う時刻を選ぶ） |
| `plan_slots` | plan_id, slot_type_id, capacity, active | プラン×開始時刻の催行設定と**基本定員** |
| `capacity_overrides` | date, plan_id, slot_type_id, capacity | **日別定員の上書き**（行があれば基本定員より優先） |
| `slot_closures` | date, slot_type_id, plan_id(null=全プラン), reason | 手動クローズ（休業日等） |
| `bookings` | id, plan_id, date, slot_type_id, agency_id(null=自社), **status(4値)**, customer_name, customer_phone, **num_adults, num_children, price_adult, price_child**, party_size, total_amount, payment_method, payment_status, notes, created_by, created_at, cancelled_at | 予約本体。単価は予約時点のスナップショット |
| `agencies` | id, name, token(unique), email, **booking_mode('realtime'/'request')**, active, notes | 代理店。booking_modeで即確定かリクエストかを制御 |
| `email_log` | id, booking_id, to, type, status, error, created_at | 通知メールの送信記録 |

- `bookings.status`: **requested（承認待ち）/ confirmed（確定）/ cancelled（キャンセル）/ denied（否認）**。cancelled/denied は論理削除で記録が残る
- `party_size` = num_adults + num_children（コアの定員計算が使う合計値。登録・変更時にアプリが整合を保証）
- `total_amount` は既定で 大人単価×大人人数＋小人単価×小人人数 を自動計算し、手修正も可能
- `payment_method` は onsite_cash / onsite_card / invoice / stripe。`payment_status` は unpaid / paid

## 4. コア在庫ロジック（コース連動 v2）

「自動クローズ/オープン」を状態として保存せず、空き状況を予約データからの**導出値**として毎回計算する。

> プランPの日付D・開始時刻Sが予約可能 ⇔
> ① Pが時刻Sに催行設定されている（plan_slots.active、plans.active）
> ② (D, S, P) が手動クローズされていない（全プラン指定のクローズも含む）
> ③ Pの (D, S) における**有効予約（requested + confirmed）**の人数合計 ＋ 新規人数 ≦ **有効定員**（capacity_overrides があればその値、なければ plan_slots.capacity）
> ④ Pが必要とするリソースを共有する**時間の重なる有効予約**が存在しない。重なり判定: 予約b（プランQ・時刻T）と候補（プランP・時刻S）は `start(T) < start(S)+duration(P)` かつ `start(S) < start(T)+duration(Q)` のとき重なる。ただし**完全に同じ枠（Q=P かつ T=S）の予約は相乗り**であり競合ではない（定員③で制御）

- **リクエスト予約は在庫を押さえる**（③④で confirmed と同格に扱う）。承認は必ず成功し、否認すれば即座に枠が戻る
- クローズ/オープンの更新処理そのものが存在しないため、状態の不整合が原理的に起きない
- **ダブルブッキング対策**: 予約登録は、上記の空き条件をWHERE句に含めた `INSERT ... SELECT ... WHERE` を1文で実行し、`meta.changes` で成否を判定する（D1の書き込みは直列化されるため同時アクセスでも安全）
- **予約の変更**（日付・時刻・人数・プランの変更）は、変更後の枠の空き条件（自分自身の予約を除外して評価）をWHERE句に含めた1文のUPDATEでアトミックに実行する。条件を満たさない場合は元の予約が無変更のまま残り、エラーを返す。予約IDは変更後も維持される。変更対象は confirmed のみ（requested は先に承認する運用）
- **承認/否認**: `requested → confirmed`（approve）/ `requested → denied`（deny）。いずれも1文UPDATE。キャンセルは requested / confirmed の両方に可能
- 管理画面からの登録も代理店と同じ空き条件に従う（強制登録機能は設けない。例外運用は手動クローズの解除やマスタ変更・日別定員で行う）
- 競合で登録できなかった場合は「この枠は直前に埋まりました」という明確なエラーを利用者に返す

## 5. 画面構成

### 管理画面（`/admin`、パスワードログイン）

- **予約台帳カレンダー（ホーム）**: 月表示。各日に開始時刻ごとの状態を色分け表示 — 空きあり/満席/連動クローズ/手動クローズ、予約人数。日クリックで日別詳細へ
- **日別詳細**: プラン×時刻の空き状況表（残席・連動クローズの原因プラン付き）、予約一覧（承認待ちには**承認/否認ボタン**）、新規予約フォーム（**大人/小人の人数・単価**）、**日別定員の上書き設定**
- **承認待ち一覧**（`/admin/requests`）: 全リクエスト予約を一覧し、承認/否認。ナビに件数バッジ
- **予約の登録・変更・キャンセル**: 電話予約等もここから手入力（在庫連動は同様に働く）
- **マスタ管理**: プラン（**所要時間・大人/小人単価**含む）/ リソース / 開始時刻 / プラン別基本定員 / 手動クローズ日
- **代理店管理**（ステップ3）: 登録、専用リンクの発行・再発行、**予約モード（即確定/リクエスト）設定**
- **CSVエクスポート**（ステップ4）: 期間指定で予約データを出力

### 代理店ページ（`/a/{token}`、ログイン不要）

- **空きカレンダー**: 向こう60日のプラン別空き状況（残席数付き）。連動クローズも自動反映
- **予約フォーム**: プラン・日付・時刻・大人/小人人数・顧客名・電話番号・備考。代理店の booking_mode が request の場合は「リクエストとして送信し、承認後に確定」の旨を表示
- **自店予約一覧**: 自店の予約の確認とキャンセルのみ（他店・自社の予約は不可視）。承認待ち/確定/否認の状態表示。キャンセル期限は当面設けない

## 6. 通知とエラー処理

- 予約成立・リクエスト受付・承認・否認・キャンセル時に自社メールへ通知。代理店にメールアドレスが登録されていれば控えも送信
- 予約確定とメール送信は分離し、送信失敗で予約が失敗しない。結果は `email_log` に記録し、失敗は管理画面に警告表示

## 8. テストと開発体制

- **TDD**: 在庫ロジックは「連動クローズ（同時刻・時間重なり）」「キャンセル/否認による自動オープン」「同時予約の競合」「定員境界（基本定員・日別上書き）」「リクエスト予約の在庫押さえ」を網羅するテストを先に書く
- **体制**: メインセッション（Fable 5）は設計・レビュー・監査および在庫コアロジックの実装を担当。画面・CRUD等の定型実装はSonnetサブエージェント、軽微な修正はHaikuに委譲する
